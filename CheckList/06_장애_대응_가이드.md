## 1. 뉴스 파이프라인 완전 실패

### 증상
```
=== 뉴스 파이프라인 완료 (crawled=0, classified=0, ...) ===
또는
뉴스 파이프라인 실패 (건너뜀): ...
```

### 진단
```bash
# 크롤링 실패 원인
grep "Step 1" logs/trading_system.log | tail -5

# 네트워크 확인
curl -s -o /dev/null -w "%{http_code}" https://www.reuters.com
curl -s -o /dev/null -w "%{http_code}" https://finance.yahoo.com

# crawl_engine 등록 확인
curl -s http://localhost:9501/api/system/features | python3 -c "
import sys, json
d = json.load(sys.stdin)
for f in d.get('features', []):
    if 'crawl' in f.get('name', '').lower():
        print(f)"
```

### 영향
- 센티넬: `news:latest_titles` 미갱신 → 뉴스 긴급도 스캔 불가 (나머지 4룰은 정상)
- 분석: `news:latest_summary` 이전 데이터 사용 (TTL 24시간)
- 텔레그램: 뉴스 알림 없음

### 대응
1. 네트워크 확인
2. 다음 60분 주기에 자동 재시도
3. 지속 실패 시 크롤러 소스 URL 점검

## 2. 로컬 MLX 모델 전체 장애

### 증상
```
로컬 분류 실패, Claude 폴백: ...  (반복)
모든 모델 분류 실패
모든 모델 빠른 분류 실패
```

### 진단
```bash
# 모델 파일 존재 확인
ls -la models/*.gguf

# GPU 메모리 확인
top -l 1 | grep PhysMem

# llama_cpp 임포트 확인
python3 -c "from llama_cpp import Llama; print('OK')"
```

### 영향
- 뉴스 분류: Sonnet 폴백 → 키워드 폴백 (비용 증가 + 품질 저하)
- 센티넬 뉴스 스캔: 헤드라인 분류 불가 (나머지 4룰은 정상)
- 번역: Bllossom도 실패할 수 있음

### 대응
1. `models/` 디렉터리에 4개 GGUF 파일 확인
2. 메모리 부족이면 다른 프로세스 종료
3. `llama-cpp-python` 패키지 재설치: `pip install llama-cpp-python --force-reinstall`

## 3. Claude API 전체 장애

### 증상
```
에이전트 실행 실패: ... (Sonnet)
Opus 리더 판단 실패: ...
에스컬레이션 Sonnet 호출 실패: ...
```

### 진단
```bash
# API 직접 테스트
python3 -c "
import anthropic
c = anthropic.Anthropic()
r = c.messages.create(model='claude-sonnet-4-20250514', max_tokens=10, messages=[{'role':'user','content':'hi'}])
print(r.content[0].text)"

# SDK 모드 확인
grep -E "mode.*sdk|AI 백엔드" logs/trading_system.log | tail -5
```

### 영향
- Layer 1: 4에이전트 전체 실패 → 스텁 결과 폴백
- Layer 2: Opus 실패 → confidence=0.3, hold 판단
- 에스컬레이션: next_cycle 폴백 (긴급 판단 불가)
- 뉴스 정밀분석: MLX 결과 유지

### 대응
1. API 키 유효성 확인 (`.env` 파일)
2. Anthropic 상태 페이지 확인
3. SDK → API 모드 전환: `curl -X POST localhost:9501/api/system/switch-backend?mode=api`
4. 장애가 지속되면 매매 수동 중지 고려

## 4. 센티넬 루프 중단

### 증상
```
센티넬 루프 종료: X회 스캔, ...
(이후 센티넬 로그 없음)
```

### 진단
```bash
# 센티넬 태스크 상태
grep -E "sentinel_task|센티넬 루프" logs/trading_system.log | tail -10

# 현재 세션 확인
curl -s http://localhost:9501/api/trading/status | python3 -c "
import sys, json
d = json.load(sys.stdin)
print(f'세션: {d.get(\"session_type\")}, 매매중: {d.get(\"is_trading\")}')"
```

### 가능한 원인
1. 세션이 `eod_processing` 또는 `closed`로 전환됨 → 정상 종료
2. 예외 발생으로 루프 탈출 → 에러 로그 확인
3. `shutdown_event` 설정됨 → 매매 중지 요청

### 대응
1. 세션 확인 — 매매 윈도우 외이면 정상
2. 에러 로그 확인
3. 매매 재시작: `curl -X POST localhost:9501/api/trading/start?force=true -H "X-API-Key: ..."`

## 5. 텔레그램 전송 실패

### 증상
```
텔레그램 전송 실패: ...
Timed out
```

### 진단
```bash
# 텔레그램 봇 토큰 확인
grep "TELEGRAM" .env | head -2

# 직접 전송 테스트
python3 -c "
import asyncio
from src.telegram.telegram_gateway import TelegramSender
async def test():
    t = TelegramSender()
    r = await t.send_text('테스트')
    print(r)
asyncio.run(test())"
```

### 영향
- 뉴스 알림 미수신 (데이터 자체는 Redis에 저장됨)
- 긴급 판단 알림 미수신 (긴급 보고서는 Redis에 저장됨)
- 매매 판단에는 영향 없음

### 대응
1. 텔레그램 봇 토큰/챗 ID 확인
2. 타임아웃 설정: connect=10s, read=15s, write=15s (이미 적용됨)
3. 재시도 3회 + 2초 딜레이 (이미 적용됨)

## 6. Redis 연결 장애

### 증상
```
Redis 캐시 읽기 실패: ...
Redis 캐시 저장 실패: ...
```

### 진단
```bash
# Redis 서버 상태
docker ps | grep redis
redis-cli PING

# Redis 메모리 사용량
redis-cli INFO memory | grep used_memory_human
```

### 영향
- 뉴스 캐시 누적 불가 → Flutter 대시보드 데이터 손실
- 센티넬: 헤드라인 읽기/watch/priority 저장 불가
- 분석: 컨텍스트 빌드 불완전 → 기본값 사용

### 대응
1. Redis 컨테이너 재시작: `docker restart stock_trading-redis-1`
2. 메모리 초과 시: `redis-cli FLUSHDB` (주의: 모든 캐시 삭제)
3. Docker 볼륨 확인

## 8. 긴급 보고서가 trading_loop에 반영 안 됨

### 증상
```
센티넬 긴급 판단 완료 (텔레그램 수신)
→ 그런데 trading_loop이 아무 행동 없음
```

### 진단
```bash
# 긴급 보고서 Redis 확인
redis-cli GET "analysis:emergency_report" | python3 -m json.tool
redis-cli TTL "analysis:emergency_report"

# trading_loop의 DecisionMaker 로그
grep "DecisionMaker" logs/trading_system.log | tail -10
```

### 가능한 원인
1. TTL 5분 만료 → trading_loop이 다음 사이클에서 읽기 전 소멸
2. trading_loop 사이클 간격이 5분 이상
3. `cache.delete("analysis:emergency_report")` 후 미소비

### 대응
1. trading_loop 사이클 간격 확인 (5분 이내인지)
2. TTL 조정 필요 시 `_EMERGENCY_REPORT_TTL` 값 증가

## 장애 심각도 판정

| 심각도 | 기준 | 행동 |
|---|---|---|
| **P0 — 즉시 대응** | 매매 중 포지션 위험 감지 불가, trading_loop 중단 | 매매 수동 중지 → 원인 해결 → 재시작 |
| **P1 — 긴급** | Claude API 전체 장애, Redis 장애 | 매매는 폴백으로 계속, 원인 병행 해결 |
| **P2 — 주의** | 뉴스 파이프라인 실패, MLX 모델 장애 | 다음 사이클 자동 복구 대기, 지속 시 개입 |
| **P3 — 관찰** | 텔레그램 실패, 번역 실패, 단일 에이전트 실패 | 로그 관찰, 자동 폴백 확인 |
