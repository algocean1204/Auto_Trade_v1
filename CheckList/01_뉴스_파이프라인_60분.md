# 뉴스 파이프라인 체크리스트 (60분 주기)

## 파이프라인 흐름도

```
Step 1: 크롤링 (30소스)
  → 🤖 없음 (HTTP)
  → Redis: news:latest_titles (TTL 5분)

Step 2: 1차 분류
  → 🤖 로컬 Qwen2.5 + Llama3.1 + DeepSeek-R1 (기사당 9회 = 3분류 × 3모델)
  → 폴백1: Claude Sonnet
  → 폴백2: 키워드 룰

Step 2.3: 유사 기사 병합
  → 🤖 없음 (ArticleMerger 룰)

Step 2.5: 정밀 분석 (impact >= 0.5)
  → 🤖 Claude Sonnet

Step 2.7: 핵심 뉴스 필터링 (impact >= 0.7)
  → 🤖 없음 (수치 필터)
  → Redis: news:key_latest

Step 2.8: 상황 추적
  → 🤖 Claude Sonnet (감지 + 평가)
  → Redis: news:situation_reports_latest

Step 3: 번역
  → 🤖 로컬 Bllossom-8B

Step 3.5: DB 저장
  → PostgreSQL articles 테이블

Step 4: 텔레그램 전송
  → 🤖 Claude Haiku (3섹션 포맷)
  → 폴백: 키워드 포맷
```

---

## Step 1: 크롤링

### 확인 로그
```bash
grep "Step 1" logs/trading_system.log | tail -5
```

### 정상 패턴
```
[Step 1] 크롤링 완료: total=XX, new=XX, failed=[], X.Xs
[Step 1] EventBus로 수집된 기사: XX건
```

### 체크 항목
- [ ] `total` 수치 > 0 인가?
- [ ] `new_count` > 0 인가? (0이면 모든 기사가 중복 — 48시간 dedup)
- [ ] `failed_sources` 가 빈 배열 `[]` 인가?
- [ ] 실행 시간이 30초 이내인가? (네트워크 지연 주의)

### Redis 확인
```bash
# 센티넬이 읽을 헤드라인 캐시
redis-cli GET "news:latest_titles" | python3 -m json.tool | head -20
redis-cli TTL "news:latest_titles"
# 기대: TTL ~300초 (5분)
```

### 이상 징후
| 증상 | 원인 | 대응 |
|---|---|---|
| `total=0` | 네트워크/소스 전체 장애 | 소스별 URL 직접 접속 테스트 |
| `new=0, total>0` | 48시간 내 모두 크롤링됨 | 정상 (첫 사이클 후 발생 가능) |
| `failed_sources` 비어있지 않음 | 특정 소스 접속 불가 | 소스 URL 확인 |
| 30초 이상 소요 | 네트워크 느림 / 타임아웃 | HTTP 타임아웃 설정 확인 |

## Step 2: 1차 분류 (로컬 MLX 3모델 앙상블)

### 확인 로그
```bash
grep "Step 2\]" logs/trading_system.log | tail -5
grep -E "로컬 분류 실패|Claude 폴백|키워드 분류" logs/trading_system.log | tail -10
```

### 정상 패턴
```
[Step 2] 뉴스 분류 완료: XX건
```

### 체크 항목
- [ ] 분류 건수가 크롤링 건수와 일치하는가?
- [ ] "로컬 분류 실패" 로그가 과도하지 않은가? (전체의 20% 이하)
- [ ] "키워드 분류" 로그가 0건인가? (3차 폴백 작동 = MLX + Sonnet 둘 다 실패)
- [ ] 분류 시간이 기사당 15초 이내인가? (기사당 9회 모델 호출)

### AI 모델 호출 추적
```bash
# MLX 앙상블 호출 확인
grep -E "qwen|llama|deepseek|분류 모델" logs/trading_system.log | tail -20

# Sonnet 폴백 호출 확인 (있으면 MLX 실패)
grep "Claude 폴백" logs/trading_system.log | tail -5
```

### 이상 징후
| 증상 | 원인 | 대응 |
|---|---|---|
| "로컬 분류 실패" 반복 | GGUF 모델 로딩 실패 / GPU 부족 | `models/` 폴더 파일 존재 확인 |
| 분류가 크롤링보다 적음 | 일부 기사 분류 중 예외 | 에러 로그 확인 |
| 전체 기사가 키워드 폴백 | MLX + Sonnet 모두 장애 | API 키/모델 파일 확인 |
| 30초/기사 이상 소요 | Metal GPU 경합 / 메모리 부족 | `top` 명령으로 메모리 확인 |

## Step 2.5: 정밀 분석 (Claude Sonnet)

### 확인 로그
```bash
grep -E "정밀 분석|refine|Claude 정밀" logs/trading_system.log | tail -10
```

### 체크 항목
- [ ] impact >= 0.5인 기사만 Sonnet으로 정밀 분석되는가?
- [ ] "Claude 정밀 분석 실패" 로그가 없는가?
- [ ] 정밀 분석 후 impact_score가 변경되었는가? (Sonnet이 재평가)
- [ ] `tickers_affected` 가 빈 배열이 아닌가?

### Redis 확인 (분류 결과)
```bash
# 분류된 전체 기사 (누적)
redis-cli LLEN "news:classified_latest" 2>/dev/null || \
redis-cli GET "news:classified_latest" | python3 -c "
import sys, json
data = json.load(sys.stdin)
print(f'총 기사: {len(data)}건')
high = [a for a in data if a.get('impact_score', 0) >= 0.7]
med = [a for a in data if 0.5 <= a.get('impact_score', 0) < 0.7]
low = [a for a in data if a.get('impact_score', 0) < 0.5]
print(f'  high: {len(high)}, medium: {len(med)}, low: {len(low)}')"
```

### 이상 징후
| 증상 | 원인 | 대응 |
|---|---|---|
| "Claude 정밀 분석 실패" | Sonnet API 장애 / 토큰 한도 | API 상태 확인, 로컬 결과 유지됨 |
| 모든 기사가 impact 0.5 미만 | MLX가 전부 low로 분류 | MLX 분류 결과 샘플 확인 |
| tickers_affected 빈 배열 | Sonnet 응답 파싱 실패 | JSON 파싱 로그 확인 |

### 확인 로그


### 체크 항목

## 파이프라인 종합 로그

### 정상 완료 패턴
```bash
grep "뉴스 파이프라인 완료" logs/trading_system.log | tail -3
```

기대 출력:
```
=== 뉴스 파이프라인 완료 (crawled=XX, merged=XX, classified=XX,
    translated=XX, persisted=XX, high_impact=XX, situations=XX, sent=True) ===
```

### 전체 소요 시간 측정
```bash
grep -E "뉴스 파이프라인 (시작|완료)" logs/trading_system.log | tail -4
```
- 정상: 3~10분 이내
- 주의: 15분 이상이면 MLX 또는 API 병목

### 동시 실행 방지 확인
```bash
grep "파이프라인 이미 실행 중" logs/trading_system.log | tail -3
```
- 이 로그가 나오면: 이전 파이프라인이 아직 실행 중 → 건너뜀 (정상 동작)
