## Metal GPU 사용 추적

### 확인 명령
```bash
# GPU 사용률 (macOS)
sudo powermetrics --samplers gpu_power -i 1000 -n 1 2>/dev/null | grep -E "GPU|Active"

# 메모리 사용량
top -l 1 | grep -E "PhysMem|Mem"

# GGUF 프로세스 확인
ps aux | grep -E "llama_cpp|gguf"
```

### 체크 항목
- [ ] 로컬 모델 4개가 GPU 메모리에 로딩되어 있는가?
- [ ] 모델 전환 시 GPU 메모리 부족이 발생하지 않는가?
- [ ] 센티넬 빠른 분류와 뉴스 파이프라인 분류가 동시에 실행되지 않는가?

### GPU 경합 시나리오
```
위험: 뉴스 파이프라인 MLX 분류 중 + 센티넬 빠른 분류 동시 실행
→ ThreadPoolExecutor(max_workers=1)로 직렬화됨 → 경합 없음
→ 단, 센티넬이 뉴스 파이프라인 완료를 기다려야 할 수 있음
```

| 모델 | 실패 시 폴백 | 영향 |

## API 동시 호출 제한 (Semaphore)

| 모델 | 최대 동시 호출 | 근거 |
|---|---|---|
| **Opus** | 4개 | 3+1 팀 = 최대 4 Opus 병렬 |
| **Sonnet** | 6개 | Layer 1 (4) + 에스컬레이션 (1) + 여유 (1) |
| **Haiku** | 4개 | 텔레그램 포맷 등 경량 작업 |

### 동시 호출 경합 시나리오
```
위험: 센티넬 emergency(Opus 4) + continuous_analysis(Opus 4) 동시
→ Semaphore(opus=4)로 센티넬이 먼저 점유 → continuous_analysis 대기
→ 센티넬 완료 후 continuous_analysis 진행 (rate limit 429 방지)
```

### 확인 로그
```bash
# Semaphore 대기 징후 확인 (Opus 응답이 비정상적으로 느린 경우)
grep -E "Opus 팀 Phase [12]|긴급 Opus" logs/trading_system.log | tail -20
```
