# 센티넬 루프 체크리스트 (2분 주기)

## 센티넬 흐름도

```
[2분 주기 — sentinel_loop.py]

_run_single_scan():
  ├── detect_anomalies() — 5개 룰 독립 실행
  │   ├── 룰 1: 가격 급변 (스캔 간 ±2%)     → 🤖 없음 (브로커 API)
  │   ├── 룰 2: VIX 급변 (±3pt)             → 🤖 없음 (FRED/Redis)
  │   ├── 룰 3: 거래량 급증 (300%+)          → 🤖 없음 (브로커 API)
  │   ├── 룰 4: 포지션 위험 (-0.7% 근접)     → 🤖 없음 (PositionMonitor)
  │   └── 룰 5: 뉴스 긴급도                  → 🤖 로컬 3모델 빠른 앙상블
  │
  ├── watch만 → Redis sentinel:watch에 누적 저장
  │
  └── urgent → 에스컬레이션
        ├── evaluate_anomaly()  → 🤖 Sonnet 1회
        │   ├── ignore → 종료
        │   ├── next_cycle → Redis sentinel:priority에 누적 저장
        │   └── emergency ↓
        │
        └── emergency_opus_judgment()  → 🤖 Opus 3+1 팀 (4회)
            ├── 공격형/균형형/보수형 3명 병렬
            └── 리더 종합
            → Redis: analysis:emergency_report (TTL 5분)
            → 텔레그램 긴급 알림
```

```
센티넬: 다음 정기 분석에 우선 반영
```

## 룰 1: 가격 급변 (스캔 간 ±2%)

### 동작 방식
- 이전 스캔 가격과 현재 가격을 비교 (전일 대비가 아님)
- 첫 스캔: 기준 가격만 저장, 트리거 안 함
- `|delta_pct| >= 2.0%` → urgent

### 체크 항목
- [ ] 감시 대상: SPY + QQQ + 보유 포지션 티커
- [ ] 첫 스캔 후 기준 가격이 설정되었는가?
- [ ] 2분 내 2% 변동은 극단적 상황 — false positive가 아닌가?
- [ ] 가격 조회 실패 시 해당 티커만 스킵되는가?

### 확인 명령
```bash
grep -E "가격 조회 실패|price_crash|price_surge" logs/trading_system.log | tail -10
```

### 이상 징후
| 증상 | 원인 | 대응 |
|---|---|---|
| 매 스캔마다 urgent | 기준 가격이 리셋됨 | SentinelState.last_prices 유지 확인 |
| 가격 조회 전부 실패 | 브로커 API 장애 | KIS API 상태 확인 |
| 2분 내 2%는 비현실적 | 정상 (매우 드문 이벤트) | 연간 2~3회 발생 가능 |

## 룰 2: VIX 급변 (±3pt)

### 동작 방식
- FRED API 또는 Redis 캐시에서 VIX 조회
- 폴백값 20.0 (±0.01 허용 오차)이면 비교 스킵
- 첫 스캔: 기준값만 저장
- `|delta| >= 3.0pt` → urgent

### 체크 항목
- [ ] VIX가 실제 값인가? (폴백 20.0이 아닌가?)
- [ ] FRED API TTL (1시간) 내에 캐시되어 있는가?
- [ ] VIX가 정확히 20.0일 때 스킵되는 것이 맞는가? (실제 VIX=20.0이면 놓침)

### 확인 명령
```bash
grep -E "VIX|vix" logs/trading_system.log | tail -10
redis-cli GET "vix:current"
```

### 이상 징후
| 증상 | 원인 | 대응 |
|---|---|---|
| "VIX 폴백값 감지" 반복 | FRED API 지속 실패 | API 키/네트워크 확인 |
| VIX 기준값이 매번 리셋 | SentinelState 재생성 | 루프 내 상태 보존 확인 |
| VIX 3pt 변동이 잦음 | 고변동성 장세 | 정상 (VIX 30+ 시 빈번) |

## 룰 3: 거래량 급증 (300%+)

### 동작 방식
- 현재 거래량 / 평균 거래량 비율 계산
- `ratio >= 3.0` → watch (urgent 아님)
- avg_volume이 0이면 스킵

### 체크 항목
- [ ] avg_volume 데이터가 유효한가?
- [ ] 장 초반 (power_open)에 volume이 비정상적으로 높지 않은가?
- [ ] watch 수준이므로 에스컬레이션 없이 Redis 저장만 됨

### 확인 명령
```bash
grep "volume_surge" logs/trading_system.log | tail -5
```

## 룰 4: 포지션 위험 근접

### 동작 방식
- 보유 포지션의 PnL% 확인
- `pnl_pct <= -0.7%` → urgent (하드스톱 -1.0% 근접)
- `pnl_pct >= take_profit × 0.8` → watch (수익목표 근접)
- VIX 폴백이면 기본 take_profit=3.0% 사용

### 체크 항목
- [ ] 포지션이 없으면 스킵되는가?
- [ ] 레짐별 take_profit이 올바르게 적용되는가?
- [ ] 하드스톱 근접 시 urgent가 발생하는가?

### 확인 명령
```bash
grep -E "position_danger|profit_target_near|하드스톱 근접" logs/trading_system.log | tail -10

# 현재 포지션 확인
curl -s http://localhost:9501/api/positions | python3 -m json.tool
```

## 룰 5: 뉴스 헤드라인 긴급도 (빠른 로컬 앙상블)

### 동작 방식
- Redis `news:latest_titles`에서 헤드라인 읽기
- SHA-256 해시로 이미 분류한 헤드라인 제외 (중복 방지)
- 신규 헤드라인만 최대 15건 분류
- `fast_local_classify()` → 3모델 앙상블 (200자 제한)
- urgent: confidence >= 0.8 필수 (watch보다 높은 기준으로 false positive 방지)
- watch: confidence >= 0.7 필수
- 금융 도메인 특화 프롬프트 사용 (crash, Fed, circuit breaker 등 구체적 키워드)
- executor 점유 시 30초 타임아웃 (뉴스 파이프라인 블로킹 방지)

### 체크 항목
- [ ] `news:latest_titles`에 데이터가 있는가? (TTL 5분)
- [ ] 신규 헤드라인만 분류되는가? (중복 로그 확인)
- [ ] 스캔당 15건 제한이 적용되는가?
- [ ] 분류 시간이 30초 이내인가? (15건 × ~2초/건)

### 확인 명령
```bash
grep -E "헤드라인 스캔|news_urgent|news_watch|헤드라인 분류 실패" logs/trading_system.log | tail -10

# 캐시된 헤드라인 확인
redis-cli GET "news:latest_titles" | python3 -c "
import sys, json
titles = json.load(sys.stdin)
print(f'헤드라인 수: {len(titles)}')
for t in titles[:5]:
    print(f'  {t[:80]}')"
```

### 이상 징후
| 증상 | 원인 | 대응 |
|---|---|---|
| "헤드라인 캐시 읽기 실패" | Redis 연결 문제 / TTL 만료 | 뉴스 파이프라인 실행 확인 |
| 신규=0 반복 | 60분 내 같은 헤드라인 | 정상 (파이프라인 60분 주기) |
| 분류 시간 60초+ | GPU 부하 / 모델 미로딩 | Metal GPU 사용률 확인 |
| urgent 과다 발생 | confidence 임계값 부족 | 0.8 이상만 urgent 확인 |
| "executor 점유로 타임아웃" | 뉴스 파이프라인 MLX 분류 중 | 정상 — 다음 스캔에서 재시도 |

### 체크 항목

## 긴급 판단: Opus 3+1 팀

### 확인 로그
```bash
grep -E "EMERGENCY|긴급 Opus|Phase [12]" logs/trading_system.log | tail -20
```

### 정상 패턴
```
센티넬: EMERGENCY — Opus 3+1 팀 긴급 판단 시작
Opus 팀 Phase 1: 3명 독립 분석 시작
Opus 팀 Phase 2: 리더 종합 시작
긴급 Opus 3+1 팀 판단: confidence=0.XX, risk=high, signals=2
```

### 체크 항목
- [ ] Opus 3명 + 리더 1명 = 총 4회 Opus 호출인가?
- [ ] 보수형이 critical risk 경고 시 hold 판정인가?
- [ ] 긴급 보고서가 `analysis:emergency_report`에 저장되었는가? (comprehensive_report 아님!)
- [ ] 텔레그램 긴급 알림이 전체 신호를 포함하는가?

### Redis 확인
```bash
# 긴급 보고서 확인 (별도 키, TTL 5분)
redis-cli GET "analysis:emergency_report" | python3 -m json.tool

# TTL 확인
redis-cli TTL "analysis:emergency_report"
```

### 텔레그램 확인
- [ ] 메시지에 "센티넬 긴급 판단 (Opus 3+1 팀)" 제목이 있는가?
- [ ] 감지 신호 전체가 나열되어 있는가?
- [ ] Opus 팀 판단 결과 (recommendations)가 있는가?
- [ ] 신뢰도/위험도가 표시되어 있는가?

### 이상 징후
| 증상 | 원인 | 대응 |
|---|---|---|
| "Opus 팀 호출 실패" | Opus API 장애 | hold/confidence=0.3 폴백 확인 |
| 텔레그램 "전송 실패" | 텔레그램 타임아웃 | 긴급 보고서는 Redis에 저장됨 |
| 긴급 보고서 5분 후 소멸 | TTL 정상 만료 | 다음 정기 분석까지 유효 |
| trading_loop이 긴급 보고서 무시 | emergency_report 키 미확인 | trading_loop.py 코드 확인 |

### 전체 센티넬 활동 타임라인
```bash
grep -E "센티넬" logs/trading_system.log | tail -30
```

### 2분 주기 확인
```bash
# 스캔 간격이 정확히 ~120초인지 확인
grep "센티넬 감지\|센티넬 루프 시작" logs/trading_system.log | \
  awk '{print $1, $2}' | tail -10
```

### 에러 누적 확인
```bash
grep -E "센티넬 스캔 실패" logs/trading_system.log | wc -l
```
- 0건: 정상
- 1~3건: 일시적 장애 (자동 복구)
- 5건+: 구조적 문제 → 코드/인프라 확인
